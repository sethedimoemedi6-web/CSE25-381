What is in a block

A block is more than just a container for transactions it is a carefully structured data package that maintains the entire chains security and integrity


Block Header (the id card)

Every block starts with a header (~80 bytes in Bitcoin)containg 6 key fields:

  • Version:indicates the protocol rules this block follows.
  • Previous Block hash:The cryptographic fingerprint(SHA-256) of the block right before it.This creates unbreakable chain
  • Merkle Root:A single hash that represents all transactions in the block (more on this below)
  • Timestamp: unix time when the block was created(miners must stay within two hours of network time)
  • Difficulty Target/Bits:the current mining difficulty encoded in compact form
  • Nonce:A 32 bit number miners change millions of time per second until the block hash meets the difficulty target

Block Body (the payload)

This contains the actual list of transactions. Bitcoin block average ~2000-~3000 txs;Ethereum contain more thanks to its account model

The Merkle Tree-Efficiency Superpower

Transactions are paired,hashed,pairted again, until one hash the merkle root remains.This allows any node to verify a single transaction inclusion in just log(n) hashes instead of scanning the entire block.Changing even one transactions would change the merkle root and therefore the entire block hash instantly breaking the chain

Real World Differences

  • Bitcoin:~1MB block size limit (SegWit increased effective size)
  • Ethereum:Dynamic gas limit (~3o million gas today)
  • Solana:No traditional blocks in the same way it uses the "slots"and accounts for massive throughput

Key takeaway

A block is a self-contained tamper evident package.The header proves the work was done ;the merkle root proves transactions are correct;the previous hash proves it belongs in the chain

How Are Transactions Authenticated

Authentication = proving i own the private key that controls these coins without reaveling the key

Public cryptography in action

Most blockchains use Elliptic Curve Digital Signature Algorithm(ECDSA)on the secp256k1 curve
  1. The sender transaction data(inputs/outputs,amount,nonce etc)
  2. Hashes that data
  3. Signs the hash with their private key produces a digital signature
  4. Broadcasts the transactions+public key+signature
verification (done by every node)Nodes re-hash the transaction data and use the senders public key to check if the signature matches.If it does->authentic.This happens in milliseconds.

Extra protections

  • Nonce/Sequence number:Prevents replay attacks
  • UTXO model(Bitcoin)vs Account model (Ethereum):UTXO requires spending whole coins ; accounts uses nonces to track order
  • Multi-Signature(multisig) and threshold signatures:Modern wallets and DAOs

Common attack vectors at this layer

  • Malleability attacks (fixed by SegWit)
  • Transaction flooding(mitigated by mempool fees)

Key takeaway

Authentication is cryptographic proof of owenership.No passwords no third party-just math

Building consensus within blockchain networks

Consensus:how thousands of independent nodes agree on which transactions happened and in what order,without a boss

Proof-of-Work(Bitcoin)

Miners compete to find nonce that produces a hash below the target.Longest chain with most cumulative work wins 51% attack possible but extremely expensive.

Proof-of-Work(Ethereum 2.0),solana,Cardono

Validators lock up (stake) native coins.The protocol randomly selects a validator to propose the next block.Others attest.Slashing penalizes bad behaviour.Much more energy efficient

Other Mechanisms

  1. Delegated Pos (Dpos): Token holders vote for delegates
  2. Proof-of-Authority(POA):Trusted validators (used in private chains)
  3. Hybrid models:Some chains combine PoS+Pow elements

Finality

  • Probabilistic(Bitcoin:~6 confirmations)
  • Deterministic/instant (many pos chains after 2/3 attestations)

Key Takeaway

Consensus is the engine of Decentralization.Different trade off energy use speed and trust assumptions

Exploring Blockchain Spectrum

Blockchains exist on a spectrum from fully public to fully private
  • Public/permissonless:(Bitcoin,Ethereum):Anyone can read write and validate decentralization
  • Public/Permissioned(Hyperledger Fabric):Only invited participants fast but less decentralized
  • Consortium:group of companies run validators together
  • Layer-2 & Sidechains : Rollups,plasma etc inherit security from layer-1 but offer different trust models

Why the spectrum matters

Public= censorship resistence+global accesibility Private=speed+regulatory compliance Most web3 projects live on public L1/l2 but use private solutions for enterprise backends

Blockchain Trilemma

Proposed by Vitalik Buterin: you can only have two of these three at once:

  1. Decentralization-No single point of failure or control
  2. Security- resistant to attacks and bugs
  3. Scalability- High transactions per second (TPS)

Examples:

  • Bitcoin:High decentralization+security ->low scalability(~7 TPS).
  • Solana:High scalabity +security->lower decentralization (fewer validators)
  • Early Ethereum:Balanced but slow.Layer-2 solutions try to solve the trilemma by moving execution off-chain while inheriting L1 security

Modern Attempts to break the trilemma :MOdular blockchains (celestia),zero-Knowledge proofs,sharding and parallel execution

Key takeaways:+

Every chain makes trade-offs.Understanding the trilemma helps you evaluate any project

Blockchain Scaling:off-chain scaling

off-chain= moving transactoins outside the main chain while keeping security guarantees

Popular Solutions

  • State channels:(lightining Network on Bitcoin):Open a channel transact instantly and privately close on-chain later
  • Rollups (optimistic& ZK on Ethereum):Batch hundreds of transactions, post compressed data + proof to L1
  • Optimistic Rollups:Assume validity ,use fraud proofs
  • ZK-Rollups:Use zero-Knowledge proofs for instant finality
  • Sidechains:independent chains with their own consensus (e.g polygon Pos)

Pros & Cons

Huge Tps boost and lower fees . Security still anchored to L1 (expect pure sidechains)

Blockchain Scaling:on-chain scaling

On-chain= improving the base layer itself

Techniques

  • Bigger blocks(Bitcoin cash)
  • Sharding (Ethereum planned darksharding):split the chain into parallel shards
  • Improved consensus+execution (solana Proof-of-history +Gulf Stream)
  • Data availality sampling and stateless clients to reduce node hardware requirements

Trade-Offs

Higher throughput but increased bandwidth/storage demands on validators ->risk of centralization

Current winner

Most L1 now combine modest on-chain improvements with aggressive off-chain(layer-2) scaling

Blockchain and WEB3 Ownership

Web3 core promise:true digital ownership

  • Self-custody:You hold the private keys ->you own the assets (not an exchange).
  • NFTs:Unique provable ownership of art,tickets,virtual land etc
  • Soundbound tokens & identity:Non-transferable on-chain credentials
  • Decentralized Autonomous Organisations(DAOs):Community-owned governance

Reality Check

  • Most users still keep their funds on centralized exchanges (custodial)
  • True ownership requires inderstanding seed phrases and gas fees
  • Regulatory uncertainity around "ownership" of digital assets

Key Takeaway:

Web3 shifts power from platforms to users but only if users actually control the keys

Web3 Business Models and Tokenomics

Common Models:

  • Play-to-earn/Move-to-earn(Axie infinity style)
  • DEFI yield farming & liquidity mining

Tokenomics 101

  • Utility:Staking Gorvenance, fee discounts across rights
  • Distribution:Team vesting, community airdrops, liquidity pools
  • Vesting& cliffs:prevent early dumps
  • Sustainable incentives :Avoid ponzi like emmision schdules

Red Flags

Hyper-inflationary tokens no clear utility founder-heavy allocations or guaranteed yields

Key Takeaway:

Great tokenomics align incentives between user, developers and investors.Bad ones create pump-and-dump schemes.
